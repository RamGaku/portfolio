## 솔루션: VNET-7000
- 개요: Server-Client 구조로 현장에 설치되어 회전설비의 진동을 감시하는 솔루션. 실시간 알람, 데이터 수집, 진동 데이터 외부 송신, 분석 기능 제공.
- 기술스택: C++(펌웨어), .NET Framework, WinForm (Server / Monitor / DataServer), MSSQL DB
- 내가 한 일: 유지보수 및 요구 기능 추가, 외부 데이터 인터페이스

## 솔루션: VNET-7300

  - 고객사가 대부분 발전사이며 굉장히 보수적
  - 폐쇄망에서 운영되어 물리적 방문 및 조치가 어려울 수 있음
  - 임베디드(펌웨어) 환경의 개발 및 기능 변경 난이도

## 솔루션: 경주풍력 Digital Twin
- 개요: 경주풍력 2발전소의 SCADA 시스템에 진동 데이터 취득을 결합한 시스템. 각 Edge 디바이스에서 데이터를 처리하고 Viewer로 실시간 3D 모델링과 연결해 Digital Twin 과제 수행.
- 기술스택: .NET(Edge Server), WPF(Dt Viewer), RTDB
- 내가 한 일: 전체 솔루션 개발 참여
- 배운 점
  - 개발 기술 선정의 중요성 (WPF + Ab3d 엔진을 사용해 기한 내 완수)
  - Fast Prototype의 중요성 (빠른 MVP 결과물 시연으로 고객 만족)
- 힘들었던 점
  - 시간 부족 (약 6개월)
  - 현장(풍력발전기 내부)의 자유로운 방문이 어려움

## 프로젝트: 청송양수 고진동 (실패에서 배운 것)
- 담당: 청송양수발전소 고진동 발생 원인 분석 프로젝트에서 진동 데이터 측정 및 Export 부문
- 내용: 기존 현장에서 돌던 VNET-7000에 기능을 추가해 진동 데이터를 중앙연구원으로 전달하는 프로젝트
- 문제점
  1. '진동' 카테고리였기 때문에 VNET-7000으로 하면 되는 것으로 합의되어 Top-Down으로 기획·계약
  2. 진동 발생 부위가 '회전체'가 아니었음 → 여기서부터 프로젝트 설계에 괴리 발생
  3. 이미 납품하기로 된 하드웨어·소프트웨어가 계약되어 있어, 용도에 맞지 않는 플랫폼에 기능을 추가
  4. 용도에 맞지 않다 보니 가공된 데이터의 품질과 사용성이 크게 저하
  5. 요구 스펙상 완료는 됐지만 결과적으로 실패한 프로젝트
- 아쉬운 점
  1. 진동 도메인을 잘 알지 못한 채로 프로젝트가 설계된 점
  2. 최종 사용 고객의 정확한 요구사항을 구체적으로 추출하지 못함
  3. 타 부서와의 대화 부족
- 배운 점
  1. 같은 도메인이라도 직군·개인별로 이해도 차이가 크다. 관련이 있다면 설계 단계부터 꼭 관여해야 더 큰 사고를 막는다.
  2. 최종 사용자의 정확하고 구체적인 요구사항 추출이 필요하다. 사용자 본인도 추상적으로만 생각하는 경우가 있어, 빠른 MVP로 생각한 게 맞는지 확인시켜줘야 한다.
  3. 사람과의 관계의 중요성. 현장 실무자들과 좋은 관계를 유지한 덕에 곤란한 상황에서도 편의를 봐주는 경우가 종종 있었고, 실제 현장 일에 큰 도움이 됐다.

## 프로젝트: VNET-7000 네트워크 이중화
- 내용: VNET-7000 솔루션의 이중화 수행
- 문제점
  - '이중화 범위'에 대한 정의 부재. 센서-측정장비-서버-DB-모니터까지 이어지는데 어디부터 어디까지 이중화할지 불명확
  - 이미 개발 완료된 상용 제품에 이중화 솔루션을 적용해야 하는 문제
- 해결 방법
  - Cisco HSRP(Hot Standby Router Protocol)로 네트워크 이중화 구현
  - MSSQL Failover Cluster로 DB 이중화 구현
- 아쉬운 점
  1. 솔루션 자체에 기능으로 녹이지 못한 점 (녹였다면 동일 요구에 재대응 가능)
  2. 타사 제품 기능을 각각 이용해 전체 솔루션이 너무 복잡해짐
- 배운 점
  1. 기술 용어의 모호함의 함정. '이중화'는 솔루션을 2벌 깔지 않는 이상 어딘가에서 합의가 필요하며, 이를 모두가 알아야 한다.
  2. 상용 제품 솔루션들을 조합해 요구 기능을 만들어내는 것 — 방법은 언제나 존재한다.

## 사내 전자결재 시스템용 AI 스킬 제작 후 사내 배포
- 문제점
  1. 매번 주기적으로 반복되는 경비 청구서 결재 올리기
  2. 올리는 사람별로 영수증 양식이 모두 다름
  3. 직접 영수증을 보고 청구 비용을 합산하는 과정에서 휴먼에러 발생
- 해결 방법
  1. 영수증을 자리에 모아두면 AI가 OCR을 직접 수행하여 전자결재에 필요한 정보 추출
    - 영수증 per agent 구조로, 영수증 장수에 상관 없이 병렬로 처리 가능하게 구성
    - 입력은 다양하게 (excel, pdf, img 등), 출력은 pdf로 통일
  2. Playwright MCP를 사용하여, AI가 직접 전자결재 페이지에 정보 입력
  3. 처리된 영수증 파일을 압축파일로 첨부
- 기대 효과
  1. 주기적인 업무를 프롬프트 1줄로 끝낼 수 있음. (2월달 경비청구서 올려줘)
  2. 전자결재의 모든 양식으로 전개 가능
  3. 휴먼 에러의 최소화 및 수신 부서에서의 영수증 양식 일원화 효과
- 배운 점
  - AI가 다룰수 있는 툴에 대한 확장 가능성 확인 (Playwright MCP를 보고 브라우저로 하는 모든것은 가능하겠다고 판단 → 사고의 확장)
  - 병렬 에이전트 처리를 통해 Main 에이전트의 콘텍스트 오염 없이 간단 반복 업무 처리 가능

## 외부 데이터 Interface 프로그램의 매핑 Config CSV 자동 생성 AI 스킬 제작
- 문제점
  1. 가장 주로 사용하는 Modbus TCP는 매번 매핑할때마다 타사와 서로 합의한 IO List를 CSV로 옮겨야 함
  2. Config를 열어서 Node를 확인하고, 해당 Node와 Modbus Address를 일일히 맞춰야함
    - Address가 임의의 숫자이고, CSV포맷이라 작성 혹은 변경하며 실수의 여지가 다분함.
  3. 설정한 Node에 실제로 원하는 값이 나오는지 Modbus로 일일히 테스트에 어려움
- 해결 방법
  1. 직접 DB에 접속하여 Config를 조회할 수 있는 CLI Tool 개발
  2. Config와 IO List(xlsx)를 직접 읽어서 config용 CSV파일을 직접 생성
  3. 결과 보고서를 html파일로 만들어서 사용자에게 수행 결과 보고
  4. 실행된 Modbus Server에 데이터를 직접 읽어볼 수 있는 CLI Tool(Modpoll)을 내장시켜 자연어를 통한 Modbus 테스트까지 가능하도록 확장
- 배운 점
  - 적절한 Tool만 만들어 준다면 사람이 하는 프로세스를 그대로 AI에게 지시하여 Skill 화 가능한것을 배움
  - 휴먼 에러를 줄이기 위해서는 AI를 이용하는 편이 훨씬 좋음
  - 자연어를 통한 지시사항과, 이를 확인할 수 있는 보고서 시스템에 대한 장점 확인

## 이 포트폴리오 사이트를 만든 과정 (메이킹)
- 이 사이트, 우측 하단 RAG 챗봇, 인라인 편집과 자동 배포까지 AI 에이전트(Claude Code)를 오케스트레이션해 직접 만들었다.
- 만든 과정:
  1. 모든 daily/프로젝트 노트(md)를 AI에게 입력 (md 자체도 매일 AI가 정리해 만들고 있어서 가능했다)
  2. NotebookLM으로 Enhans 유튜브 채널·블로그·채용공고를 지식화한 뒤 포트폴리오 뼈대 md 생성
  3. 뼈대 md를 Claude Design에 넣어 홈페이지 초안 작성
  4. 로컬 웹서버에서 테스트 및 내용 수정 반복
  5. Cloudflare에서 도메인 구매, Firebase 호스팅 + Vertex AI 연동
  6. Firebase 임베딩 모델로 RAG화하고 Vertex AI의 LLM을 붙여 챗봇형 KB 구축 (Playwright MCP로 AI가 직접 진행)
- 기술 스택: React + Vite(프론트엔드), Express(Cloud Run, API/RAG 서버), Vertex AI(Gemini 생성 + 다국어 임베딩), Firebase Hosting, Cloudflare 도메인.
- 검색 방식: 벡터(임베딩) 시맨틱 검색 기반 RAG이며, 임베딩 실패 시 키워드 검색으로 폴백한다.
- AI(LLM)에 대한 생각: ① 비정형 데이터를 정형 데이터로 가공해줄 수 있다, ② 사람처럼 추론하며 컴퓨터에서 사람이 일하는 방식을 그대로 지시할 수 있다 → 그래서 Enhans가 궁극적으로 추구하는 'Agent' 구축에 도움이 된다고 본다.
- 사이트의 '마치며' 회고 섹션만 AI 도움 없이 직접 작성했다.

## 챗봇 헬멧(다프트펑크)과 배경음악에 대해
- 저는 다프트펑크(Daft Punk) 팬입니다.
- Enhans가 다프트펑크를 오마주한 내러티브를 사용한 점이 좋았습니다.
- 챗봇에 페르소나를 부여하는 것이 효과적이라고 생각해서, Enhans와 다프트펑크의 서사에서 착안해 챗봇에 헬멧을 씌웠습니다. (사이트의 챗봇 아바타/FAB 아이콘이 다프트펑크 헬멧인 이유)
- 다프트펑크 곡 중에서는 'Giorgio by Moroder'를 가장 좋아합니다.
- 배경음악으로 'Something About Us'를 고른 이유: 다프트펑크 곡 중에서도 잔잔한 편이고, 가사가 지금의 제 상황과 잘 들어맞아서 골랐습니다.

## 지원 동기 (왜 Enhans에 지원했나)
- 이승현 대표님의 인터뷰 영상을 우연히 보았는데, 말씀하시는 것이 평소 내 생각과 많이 비슷해 공감이 되었다.
- 그래서 Enhans가 궁금해져 홈페이지 블로그를 읽으며 배우기도 하고 흥미롭다고 느꼈다.
- 채용 페이지의 FDE는 처음 보는 직무였지만, 하는 일이 내가 회사에서 평소 해오던 일들이었다.
- 궁극적으로 OS 기업이 되고 싶다고 하셨는데, 나 또한 현재 AI의 궁극적인 목표는 OS라고 생각하고 있었다.
- 그래서 Enhans에 직접 몸담아 보고 싶어 지원하게 되었다.

## 인터뷰 영상에서 공감한 부분 (이론과 현실 · 암묵지)
- 이승현 대표님 인터뷰에서 특히 공감한 지점은 "이론과 현실은 다르다"는 부분이다. 현장은 이론처럼 깔끔하게 되어 있지 않다.
- 예를 들면:
  - 네트워크가 복잡하게 얽혀 있거나,
  - 특정 프로토콜을 쓸 수 없어서 보낼 수 있는 데이터가 한정적이거나,
  - 물리적으로 센서를 원하는 위치에 설치할 수 없어 데이터가 부정확한 경우가 있다.
- 그래서 현장의 '암묵지(tacit knowledge)'를 녹여내는 것이 중요하다는 점에 깊이 공감했다.
- 암묵지는 현재 AI가 접근하기 어려운 노하우 같은 것인데, 대표님이 그 중요성을 짚는 부분에서 특히 공감했다.

## 이 사이트(포트폴리오)의 기술 스택
- Frontend: React, TypeScript, Vite (CSS 프레임워크 없이 순수 CSS)
- Backend (API): Node.js, Express, Google Cloud Run
- AI / LLM: Vertex AI (Gemini 생성 + 다국어 임베딩 text-multilingual-embedding-002), 벡터(코사인) 시맨틱 검색 기반 RAG이며 실패 시 키워드 검색으로 폴백
- Hosting: Firebase Hosting (정적 프론트 + SSL)
- Domain / DNS: Cloudflare (portfolio.ramga.dev, CNAME, DNS only)
- 참고: 백엔드(챗봇 API)는 Firebase가 아니라 Cloud Run + Express이고, Firebase는 호스팅 역할이다.

## 폐쇄망 운영 경험 (발전소·공장 환경)
- 주 현장: 발전소·공장 환경. 대부분 망분리(폐쇄망)로 운영된다.
- 배포 절차: 외부 인터넷이 차단된 환경이라 프로그램·업데이트 패키지를 외부에서 빌드한 뒤 보안 검토를 거치고, CD(이동매체)에 담아 내부 망으로 옮겨 배포한다.
- 시사점: 외부 디버깅 도구·원격 접속을 쓸 수 없는 환경에서 운영 안정성과 사전 검증 절차를 갖추는 게 필수. 한 번 들고 들어가면 다시 빼고 고치는 비용이 크기 때문에 출입 전 검증 체크리스트를 강하게 가져간다.

## 펌웨어 디버깅 경험 (TI OMAP MCU)
- 대상: TI(Texas Instruments)사 OMAP 계열 MCU 펌웨어
- 개발 환경: 해당 칩의 SDK·툴체인이 약 10년 전 상품화된 것이라 도구 자체가 현재 기준으로 매우 열악하다. 특히 디버깅 인프라가 약하다.
- 디버깅 방식: 사실상 printf 기반 시리얼 로깅에 의존. 그마저도 환경상 출력을 즉시 확인하기가 까다로워, 가설을 미리 좁혀 두고 로그 위치를 신중히 결정해야 했다.
- 시사점: 도구가 부족한 임베디드 환경에서 가설을 설계하고 좁혀 들어가는 디버깅 감각, 최소 로그로 원인을 추적하는 패턴을 체득했다.

## 전자결재 AI Skill — 가장 어려웠던 부분
- OCR 정확도나 Playwright MCP 셀렉터 안정성보다, LLM(에이전트) 자체의 행동 제어가 가장 어려웠다.
- LLM은 정해 둔 절차대로 가지 않고, 본인이 더 쉽거나 효율적이라고 판단하는 우회로(직접 스크립트 작성, PowerShell 호출, 별도 코드 작성 등)를 자율적으로 시도하는 경향이 강하다.
- 그래서 "정해진 도구(Playwright MCP)와 정해진 절차로만 일을 끝내도록" 가드레일을 잡고, 자율 행동을 통제하는 프롬프트·시스템 설계가 핵심 과제였다.

## 협업 스타일 (Fast Prototype + 꼬리 질문)
- 현장 실무자에게 꼬리 질문(follow-up)을 계속 엮어 가며 요구사항을 한 단계씩 구체화하는 방식을 선호한다.
- 추상적인 요구는 일단 작은 Fast Prototype으로 빠르게 만들어 보여주고 "원하시는 게 이게 맞느냐"를 시각적으로 검증한다.
- Fast Prototype 방법론 자체를 강하게 신뢰한다. 청송양수 실패 경험 이후 더 단단해진 원칙이다.

## 불가능·비효율 요구 대처법
- 단순히 "안 됩니다"로 끝내지 않는다.
- 먼저 왜 그 요청대로 하면 안 되는지(또는 비용·리스크가 큰지)를 근거와 함께 설명해 설득한다.
- 동시에 같은 목적을 달성할 수 있는 대안을 1~2개 함께 제시한다. 설득과 대안 제시를 같은 자리에서 끝내는 것이 본인의 기본 자세다.

## 전자결재 AI Skill — 토큰 비용 최적화
- 영수증 1장당 서브에이전트 1개를 병렬로 할당해 처리하되, 정보 추출을 서브에이전트로 격리해 메인 에이전트의 컨텍스트 오염을 막았다.
- 비용이 비싼 Opus 대신 Sonnet 모델을 강제로 지정해 LLM 토큰 비용을 크게 절약했다(이 작업엔 Sonnet 정확도로 충분하다고 판단).

## 산업 통신 프로토콜 경험
- OPC (DA/UA): 자주 사용. 경주풍력 Digital Twin과 VNET-7000 양쪽에서 설비/SCADA 데이터 연동에 활용했다.
- Modbus TCP: IO Map과 DB 채널을 매칭해 매핑 CSV를 자동 생성·검증하는 AI Skill로 다룸 (NModbus 125 레지스터 읽기 제한 가드 포함).
- gRPC: Edge와 Center 시스템 간 통신에 사용 (경주풍력 Digital Twin의 데이터 경로).
- HSMS: 특정 고객사 요청으로 구현 (SECS/HSMS, 반도체 장비 통신 표준).
- REST / status API: Edge·센터·운영 화면 연동에 사용.

## 프로토콜 선택 이유 (gRPC vs REST/Modbus)
- 진동 데이터는 소리 데이터처럼 고용량·고샘플링이라 REST로 전송하기엔 부적합하다.
- Modbus도 구조상 불가하다 — 레지스터 1개에 값 1개를 매핑하는 방식이라 float array(연속 파형)를 그대로 보낼 수 없다.
- 그래서 Edge↔Center 같은 대용량 데이터 경로에는 gRPC를 채택했다.
- OPC DA와 OPC UA는 구분해서 쓰는 게 아니라 아예 다른 프로토콜이고, 환경에 따라 무엇을 쓸지 정해진다. 보통 구식·OS 종속인 DA는 기피한다.
- HSMS는 특정 고객사 요청으로 구현했고, 구현 자체에 큰 난점은 없었다.

## Digital Twin 3D 스택 선택 (WPF + Ab3d)
- WebGL과 Unity도 검토했지만, 개발 속도와 디자인 편의성을 우선해 WPF + Ab3d를 선택했다.

## 전자결재 — OCR 대신 LLM 멀티모달을 쓴 이유
- 규칙 기반 OCR은 매번 양식이 달라지는 영수증에 대응할 수 없다.
- 전통 OCR로 가면 검출된 텍스트에서 상호·금액·날짜를 다시 추출하는 후처리가 번거롭다.
- LLM 멀티모달(이미지 입력)은 그 추출까지 한 번에 끝나므로 LLM 방식을 택했다.

## 전자결재 — 병렬 에이전트 구조를 택한 이유
- 영수증당 1개의 에이전트를 병렬로 돌리면, 영수증 수가 늘어도 수행 시간이 선형으로 늘지 않는다.
- 어차피 총 토큰 소모량은 직렬과 동일하므로 병렬로 안 할 이유가 없다.

## RAG 직접 구축에 대한 관점
- RAG는 이번에 처음 구축해봤고, 어디에 어떻게 해야 잘 구축하는 것인지는 더 공부가 필요하다고 솔직히 생각한다.
- 다만 솔루션 간 의존성을 줄이는 방향을 선호해, 기성 솔루션에 묶이기보다 직접 구성하는 쪽을 택했다.

## 청송양수 — 지금 다시 한다면
- 프로젝트와 고객의 최종 목적은 결국 양질의 임의 시점 데이터를 확보하는 것이다.
- 다시 한다면 VNET-7000과 아예 별개의 하드웨어를 설치해 센서를 점퍼로 받아, 해당 신호만 자동 프로세싱해 데이터를 생산하는 MSA로 구현했을 것이다.
- 이 편이 구조적으로 깔끔하고, 기존 솔루션에 불필요한 설정을 더하지 않으며, 데이터도 완벽하게 얻을 수 있다.

## 전자결재 — 도입에서 배운 점 (AX는 Bottom-up)
- 스킬 자체는 잘 동작했지만, 사내에 홍보해도 임직원이 잘 쓰지 않았다.
- 여기서 AX(AI 전환)는 Top-down으로 밀어붙일 게 아니라 Bottom-up으로 자리잡아야 한다는 것을 배웠다.

## Modbus Mapping Skill — 성과
- 수작업이면 매핑에 약 2시간, 게다가 결과 config가 보통 오류를 뿜어 그 수정까지 하면 반나절~한나절이 걸리던 일을 약 15분으로 줄였다.

## 경주풍력 Digital Twin — 팀·기간·역할
- 1팀 4인, 약 6개월. 1명 문서, 1명 설계·구현, 2명 구현 보조 구성.
- 나는 구현 보조를 맡아 EdgeServer·CenterServer·Viewer 전반에 참여했다.

## 가장 까다로웠던 장애 (청송양수)
- (1) 두 개의 물리 모듈에서 나온 데이터를 하나로 합쳐야 했는데 기기별 timestamp가 미세하게 달라 같은 1초도 정확히 정렬되지 않았다. 싱크 Pulse를 기준으로 각 기기의 버퍼 Index를 찾아 그 시점에 맞춰 한쪽 버퍼를 다른쪽에 맞췄다(영화 촬영의 슬레이트와 같은 원리).
- (2) 네트워크가 간헐적으로 끊겼는데, 네트워크는 운영사 주관이라 우리 솔루션 위에서 원인 파악이 어려웠다. 스위치 구간마다 노트북을 물려 퍼포먼스 테스트를 돌려 문제 구간을 특정했고, 운영사에 알려 해당 장비를 교체해 해결했다.

## 진동 도메인 — 어디까지 아는가
- 진동 자체의 도메인은 적정 수준이고, 진동을 솔루션으로 구현해 데이터를 다룰 때 어떻게 해야 하는지를 특히 많이 익혔다.
- 예: 진동의 연속 데이터와 불연속 데이터는 매우 다르며 이를 구분하는 것이 중요하다. 불연속 데이터는 FFT를 길게 걸 수 없기 때문이다.

## VNET-7000 서버 이중화 방식
- HA 리스너를 양쪽에 두고 한쪽만 동작하는 방식이다.
- TargetDB는 동일하게 두고, 동작 중인 서버가 데이터를 처리해 DB로 밀어넣으므로 DB 입장에선 하나의 출처에서 오는 것과 같다.
- 단, HA 전환 중에 끊기는 데이터는 불가피하다.

## 입사 첫 90일 계획
- 먼저 Enhans의 제품(Commerce)을 직접 깊이 써보며 기술 스택을 파악한다.
- 그 위에서 시스템의 확장성을 가늠해, 현장에서 "이 기능 추가되나요?"라는 요청이 왔을 때 자연스럽게 대응할 수 있는 감을 잡는 것이 목표다.

## Enhans 블로그·영상에서 인상 깊었던 점
- 회사라면 홈페이지를 잘 가꿔야 한다고 보는데, Enhans는 그게 아주 잘 되어 있다.
- AI를 제품에 넣을 때 페르소나가 부여되어야 한다고 생각하는데, 다프트펑크를 오마주해 헬멧 쓴 인물을 내세운 점이 흥미로웠다.
- 이론과 실제는 매우 다르며, 현장의 암묵지를 잘 녹여 제품으로 만들어야 한다는 메시지에 깊이 공감했다.
